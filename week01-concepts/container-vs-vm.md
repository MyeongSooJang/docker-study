# Container vs VM vs JVM

## 📚 세 가지 기술의 역할

```
문제: 환경이 다르면 같은 코드도 다르게 동작한다

세 가지 해결책:

1️⃣ JVM (Write Once, Run Anywhere)
   목표: Java 코드를 OS에 상관없이 실행
   방식: Java 바이트코드 → JVM이 OS에 맞게 해석 실행
   한계: JVM 버전, 라이브러리, 설정은 환경에 따라 다름

   예: java -jar app.jar (Java 8)
       java -jar app.jar (Java 17) ← 버전이 다르면 동작 다름!

2️⃣ VM (Virtual Machine)
   목표: 여러 OS를 한 서버에서 동시 실행
   방식: Hypervisor가 각 VM에 독립적인 OS를 제공
   장점: 완벽한 격리
   단점: OS 전체를 복제하므로 무겁고 느림 (몇 GB, 부팅 분 단위)

3️⃣ Container (Docker)
   목표: 환경 전체를 코드로 정의해서 어디서나 같게 실행
   방식: 호스트 OS 커널은 공유하되, 파일시스템/네트워크/프로세스만 격리
   장점: 가볍고 빠르면서도 완벽한 환경 재현 (MB 단위, 초 단위)
```

## 🎯 왜 Container를 쓰는가?

| 상황 | JVM만 | VM | Container |
|------|-------|-----|-----------|
| 개발팀 노트북 | ✅ 잘 됨 | 무거움 | ✅ 완벽 |
| CI/CD 빌드 서버 | 🟡 버전 관리 복잡 | 느림 | ✅ 빠름 |
| 운영 서버 배포 | 🔴 환경 차이로 버그 | 비쌈 | ✅ 효율적 |
| 여러 앱 한 서버 | 🟡 포트/의존성 충돌 | 격리 완벽 | ✅ 격리 + 가벼움 |
| 스케일링 | 🟡 수동 설정 | 느려서 어려움 | ✅ 자동화 가능 |

## 🔍 핵심 개념 비교

| 항목 | VM | Container | 왜 이게 중요한가? |
|------|-----|-----------|-----------------|
| 격리 단위 | OS 전체 (무거움) | 프로세스 (가벼움) | 서버 리소스 효율 |
| 부팅 시간 | 수 분 | 수 초 | 배포 속도 |
| 용량 | GB 단위 | MB 단위 | 저장공간, 네트워크 전송 |
| 커널 | 독립된 커널 보유 | 호스트 커널 공유 | 리소스 오버헤드 |
| 이식성 | 낮음 (환경 차이) | 높음 (어디서나 같음) | 개발→운영 환경 동일성 |

## 구조 비교

### 🔴 VM 방식 (무겁지만 완벽한 격리)
```
┌─ Host OS (Linux/Windows) ──────────────┐
│ ┌─ Hypervisor (자원 분할) ────────────┐ │
│ │                                      │ │
│ │  ┌─ VM1 ─────────────┐               │ │
│ │  │ Guest OS (Linux)  │ (700MB~GB)   │ │
│ │  │ Java + Libraries  │               │ │
│ │  │ App.jar           │               │ │
│ │  └───────────────────┘               │ │
│ │                                      │ │
│ │  ┌─ VM2 ─────────────┐               │ │
│ │  │ Guest OS (Linux)  │ (700MB~GB)   │ │
│ │  │ Java + Libraries  │               │ │
│ │  │ App.jar           │               │ │
│ │  └───────────────────┘               │ │
│ │                                      │ │
│ └──────────────────────────────────────┘ │
│                                          │
│ ⏱️ 부팅 시간: ~2분                       │
│ 💾 용량: VM 하나에 2GB 이상 (3개면 6GB+) │
└──────────────────────────────────────────┘
```

### 🟢 Container 방식 (가볍고 빠른 격리)
```
┌─ Host OS (Linux) + Kernel ──────────────────┐
│ ┌─ Docker Engine (프로세스 격리) ─────────┐ │
│ │                                          │ │
│ │  ┌─ Container1 ──────┐                   │ │
│ │  │ /app (격리된 fs)   │ (50~100MB)      │ │
│ │  │ Java + Libraries   │ (공유/레이어)    │ │
│ │  │ App.jar (프로세스)  │ (호스트 커널 사용)│ │
│ │  └────────────────────┘                   │ │
│ │                                          │ │
│ │  ┌─ Container2 ──────┐                   │ │
│ │  │ /app (격리된 fs)   │ (50~100MB)      │ │
│ │  │ Java + Libraries   │ (공유/레이어)    │ │
│ │  │ App.jar (프로세스)  │                 │ │
│ │  └────────────────────┘                   │ │
│ │                                          │ │
│ └──────────────────────────────────────────┘ │
│                                              │
│ ⏱️ 부팅 시간: 초 단위                        │
│ 💾 용량: 기본 이미지 100MB, 컨테이너당 +MB  │
└──────────────────────────────────────────────┘

💡 핵심: 호스트 OS의 커널을 공유하므로
   새로운 OS를 부팅할 필요 없음!
```

## 💼 면접 답변 템플릿

**Q1. VM과 Container의 차이를 설명해 주세요.**

A. VM은 Hypervisor 위에 **완전한 Guest OS를 별도로 실행**하는 방식입니다.
   따라서 OS 부팅 시간(분 단위)과 리소스(GB 단위)가 많이 필요합니다.

   반면 Container는 **호스트 OS의 커널을 공유**하고 파일시스템과 네트워크만 격리합니다.
   부팅 시간이 초 단위이고 리소스(MB 단위)도 적게 써서, 한 서버에서 더 많은 애플리케이션을 실행할 수 있습니다.

**Q2. 그럼 JVM과 Container는 뭐가 다른가요?**

A. JVM은 Java 코드의 **이식성을 제공**합니다. 같은 .jar 파일이 다양한 OS에서 실행됩니다.
   하지만 JVM 버전, 라이브러리 버전, 환경변수 등이 다르면 같은 코드도 다르게 동작할 수 있습니다.

   Container는 JVM 자체를 포함해서 **전체 환경(JVM + 라이브러리 + 설정)을 패키징**합니다.
   그래서 개발 환경 = 운영 환경이 되어 "내 컴에선 되는데 서버에선 왜 안 돼?"라는 상황을 없애줍니다.

## 핵심 요약

| 키워드 | 설명 |
|--------|------|
| 격리 | 다른 컨테이너와 독립된 환경 |
| 이미지 | 컨테이너 실행을 위한 읽기 전용 템플릿 |
| 컨테이너 | 이미지를 실행한 인스턴스 (프로세스) |
| Docker Engine | 컨테이너를 관리하는 런타임 |
